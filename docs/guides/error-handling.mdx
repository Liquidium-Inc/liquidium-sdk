# Error Handling

Handle validation before you call state-changing methods. Handle SDK errors at the boundary where your app can show a user-facing message or retry.

## LTV Validation

```ts
const ltv = client.quote.calculateLtv(request, pools, prices);

if (ltv.validationErrors.length > 0) {
  return {
    ok: false,
    message: ltv.validationErrors.map((error) => error.message).join(" "),
  };
}
```

## SDK Errors

```ts
import { LiquidiumError } from "@liquidium/client";

try {
  const loan = await client.instantLoans.get({ ref });
  return loan;
} catch (error) {
  if (error instanceof LiquidiumError) {
    throw new Error(error.message);
  }

  throw error;
}
```

Use the exported `LiquidiumErrorCode` values when your UI needs different copy for timeout, transport, validation, or protocol failures.

## External Address Validation

Lending outflow and instant-loan destination fields accept external addresses for the pool or loan asset:

| Asset target | Accepted format |
| --- | --- |
| BTC mainnet | A valid mainnet Bitcoin address |
| ETH stablecoins | A valid EVM address |

The SDK validates BTC mainnet addresses with Bitcoin mainnet rules and leaves the address unchanged. It validates EVM addresses with `viem` and stores the checksummed `getAddress(...)` form.

Invalid lending `receiverAddress`, instant-loan `borrowDestination`, instant-loan `refundDestination`, and contract-interaction `account` values throw `LiquidiumError` with `LiquidiumErrorCode.INVALID_ADDRESS`. Catch that code at the form boundary and ask the user to enter an address for the selected asset and chain before retrying the flow.
